# Decisões técnicas a seguir
- **Pacotes**: respeitar a estrutura definida em `CLAUDE.md`/`escopo-todo.md`. Enums em `entity` ou `entity.enums` conforme o escopo definir; entidades em `entity`
- **ID**: `Long` autoincrement com `@GeneratedValue(strategy = GenerationType.IDENTITY)` salvo se o escopo disser outra coisa
- **Datas**: `LocalDateTime` (não `Date`, não `Timestamp`)
- **Timestamps automáticos**: usar `@PrePersist` e `@PreUpdate` (preferir sobre `@CreationTimestamp`/`@UpdateTimestamp` por ser mais explícito e não depender do Hibernate)
- **Validações Bean Validation**: aplicar nos campos da entidade conforme regras do escopo (`@NotBlank`, `@Size`, `@NotNull`, etc.) — usar `jakarta.validation.*` se Spring Boot 3+, `javax.validation.*` se Spring Boot 2.x
- **Enum no banco**: `@Enumerated(EnumType.STRING)` sempre — nunca ORDINAL
- **`equals`/`hashCode`**: NÃO sobrescrever nesta etapa (deixar default do `Object`). Se for absolutamente necessário, usar apenas o ID com checagem de null e tipo via `instanceof`/`getClass`, e justificar em comentário
- **Lombok**: usar SOMENTE se já estiver declarado em `CLAUDE.md` ou `pom.xml`. Caso contrário, escrever getters/setters/construtores explicitamente
- **Construtores**: construtor sem argumentos (exigido por JPA) + construtor com campos obrigatórios. NÃO criar construtor com todos os campos incluindo ID e timestamps
- **Anotações da tabela**: `@Table(name = "...")` e `@Column(name = "...", nullable = ..., length = ...)` explícitos, nomes em snake_case no banco

# Estilo de código
- Java 11 idiomático, sem usar features de Java 14+ (`record`, `var` em campos, etc.)
- JavaDoc curto apenas em classes e em campos não-óbvios. Sem JavaDoc gerado por hábito ("Gets the id" em getter de id)
- Ordem dos elementos na classe: campos → construtores → callbacks JPA (`@PrePersist`, `@PreUpdate`) → getters/setters
- Imports organizados, sem wildcard (`import java.util.*`)
