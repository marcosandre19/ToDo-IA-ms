# Objetivo
Implementar o repository da entidade `Tarefa` usando Spring Data JPA.

# Decisões técnicas (seguir exatamente)

## Estrutura
- **Interface**, nunca classe (Spring Data gera a implementação em runtime)
- **Nome**: `TarefaRepository`
- **Pacote**: conforme `escopo-todo.md`/`CLAUDE.md` (provavelmente `repositorio`)
- **Estende**: `JpaRepository<Tarefa, Long>` — engloba `CrudRepository` e `PagingAndSortingRepository` e adiciona métodos JPA-específicos
- **Sem anotação `@Repository`**: redundante em interfaces que estendem `JpaRepository`, adiciona ruído visual
- **Sem implementação concreta**: não criar `TarefaRepositoryImpl` salvo se o escopo exigir comportamento customizado impossível de declarar como query method

## Métodos
- **NÃO declarar métodos básicos do CRUD** (`save`, `findById`, `findAll`, `deleteById`, `existsById`) — já vêm do `JpaRepository`, redeclarar é poluição
- **Declarar APENAS query methods que o escopo `escopo-todo.md` exigir explicitamente** (filtros, buscas específicas, contagens). Se o escopo não pede nenhum, a interface fica vazia — isso é correto
- Query methods devem usar **derivação por nome** (`findByStatus`, `findByTituloContainingIgnoreCase`, `countByStatus`, etc.). Não usar `@Query` nesta etapa
- Nomes em pt-br para a parte do domínio (`findByStatus`, `findByTituloContendo`) — atenção: o **prefixo do método** (`findBy`, `countBy`, `existsBy`, `deleteBy`) é palavra-chave do Spring Data e fica em inglês; só o **nome dos campos** que entra em pt-br, e tem que **bater exatamente com o nome do atributo na entidade `Tarefa`**

Exemplo (se o escopo pedir busca por status):
```java
List<Tarefa> findByStatus(StatusTarefa status);
```

O `findBy` é Spring Data; `Status` casa com o atributo `status` da entidade.

## Estilo de código
- Java 11 idiomático
- JavaDoc curto e em pt-br **apenas** se o método tiver comportamento não-óbvio. Query methods derivados não precisam de JavaDoc — o nome já é a documentação
- Imports sem wildcard
