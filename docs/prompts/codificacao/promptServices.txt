# Decisões técnicas (seguir exatamente)

## Estrutura
- **Sem interface separada**: classe concreta `TarefaService` direto, anotada com `@Service`
- **Pacote**: conforme `escopo-todo.md`/`CLAUDE.md` (provavelmente `servico`)
- **Injeção de dependência**: por **construtor** (não usar `@Autowired` em campo, não usar setter injection). Construtor único permite o Spring injetar automaticamente sem anotação
- **Campos do repository**: `private final TarefaRepository tarefaRepository`

## Transações
- `@Transactional(readOnly = true)` na **classe** (default seguro para leitura)
- `@Transactional` (sem `readOnly`, ou seja, escrita) nos métodos que **modificam** estado: criar, atualizar, apagar
- Métodos de leitura (buscar, listar) herdam o `readOnly` da classe — não precisa anotar de novo
- Importar `org.springframework.transaction.annotation.Transactional` (NÃO `javax.transaction.Transactional` — comportamento diferente)

## Tipos de retorno e parâmetros
- Service retorna **entidade `Tarefa`**, `List<Tarefa>` ou `void`
- **NÃO retornar `Optional`** — o controller precisa de 404 padronizado, e quem decide isso é o service via exception
- **NÃO criar DTOs nesta etapa** — DTOs de request/response são responsabilidade do controller (etapa seguinte). Service trabalha com entidades
- Service recebe parâmetros como entidade ou tipos primitivos (`Long id`, `Tarefa novosDados`)

## Operações

### Criar
- Assinatura: `public Tarefa criar(Tarefa tarefa)` (ajuste o nome conforme convenção do escopo)
- Recebe uma `Tarefa` sem ID e sem timestamps preenchidos
- Persiste via `tarefaRepository.save(...)`
- Retorna a `Tarefa` salva (com ID e `dataCriacao` populados pelo `@PrePersist`)
- Não validar manualmente campos obrigatórios — Bean Validation já cuida disso na entrada do controller na próxima etapa

### Atualizar (substituição total — PUT)
- Assinatura: `public Tarefa atualizar(Long id, Tarefa novosDados)`
- Busca a entidade existente via `findById`; se não encontrar, lança `TarefaNaoEncontradaException`
- **Copia os campos atualizáveis** da `novosDados` para a entidade gerenciada — NÃO fazer `save` num objeto novo com mesmo ID (isso quebra o `dataCriacao` original e é antipattern)
- **Não copiar**: `id`, `dataCriacao`
- **Copiar**: campos de domínio conforme a entidade `Tarefa` (titulo, descricao, status, etc.)
- O `@PreUpdate` da entidade cuida de atualizar `dataAtualizacao` automaticamente
- O `save` final é opcional (entidade gerenciada dentro de transação faz dirty checking) — mas chamar `repository.save(tarefa)` explicitamente é mais legível e seguro
- Retorna a entidade atualizada

### Buscar por ID
- Assinatura: `public Tarefa buscarPorId(Long id)`
- Usa `tarefaRepository.findById(id)` e `.orElseThrow(() -> new TarefaNaoEncontradaException(id))`
- Retorna a `Tarefa` ou propaga a exceção

### Listar
- Assinatura: `public List<Tarefa> listar()` (sem parâmetros, ou conforme escopo)
- Retorna `tarefaRepository.findAll()`
- Se o escopo definir filtros (status, etc.), criar overload ou método específico — usar query method já declarado no `TarefaRepository`
- **Se o escopo não menciona paginação, NÃO implementar paginação** nesta etapa (sem `Pageable`, sem `Page<Tarefa>`)

### Apagar
- Assinatura: `public void apagar(Long id)`
- Verifica existência primeiro: `if (!tarefaRepository.existsById(id)) throw new TarefaNaoEncontradaException(id)`
- Em seguida: `tarefaRepository.deleteById(id)`
- Sem retorno

## Estilo de código
- Java 11 idiomático
- JavaDoc curto e em pt-br nos métodos públicos descrevendo: o que faz, parâmetros relevantes, exceção lançada se houver. Não comentar o óbvio
- Comentários internos em pt-br, apenas onde agregam valor
- Imports sem wildcard
- Sem `@Autowired` em campo, sem setter injection
- Sem Lombok salvo se já estiver declarado no projeto
- Ordem dos elementos na classe: campos → construtor → métodos públicos (na ordem do CRUD: criar, buscar, listar, atualizar, apagar) → métodos privados auxiliares se houver
