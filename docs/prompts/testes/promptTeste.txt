# Decisões técnicas (seguir exatamente)

## Stack de testes
- **JUnit 5** (já incluído no `spring-boot-starter-test`)
- **Mockito** (já incluído) com `@ExtendWith(MockitoExtension.class)`
- **AssertJ** (já incluído) — preferir `assertThat(...)` em vez de `assertEquals(...)`
- **NÃO usar `@SpringBootTest`** — service não precisa de contexto Spring para teste unitário
- **NÃO usar banco real ou H2** — repository é mockado

## Estrutura da classe de teste

```java
@ExtendWith(MockitoExtension.class)
class TarefaServiceTest {

    @Mock
    private TarefaRepository tarefaRepository;

    @InjectMocks
    private TarefaService tarefaService;

    // métodos auxiliares de fixture (private)
    // testes (@Test)
}
```

- Sem campos estáticos de fixture (cada teste cria seus próprios objetos via método auxiliar)
- Métodos auxiliares (`tarefaValida()`, `tarefaPersistida(Long id)`) ajudam a evitar repetição

## Convenção de nomes
- Padrão: `deve<Comportamento>Quando<Contexto>` em pt-br, sem acentos no nome do método (Java permite acentos mas geram ruído em logs e ferramentas)
- Exemplos:
  - `deveCriarTarefaComSucesso`
  - `deveRetornarListaVaziaQuandoNaoExistemTarefas`
  - `deveLancarExcecaoQuandoBuscarTarefaInexistente`
  - `devePreservarDataCriacaoAoAtualizarTarefa`
  - `deveLancarExcecaoQuandoApagarTarefaInexistente`
- **`@DisplayName`** em pt-br **com** acentos é bem-vindo em testes-chave para legibilidade no relatório (ex.: `@DisplayName("Deve preservar dataCriacao ao atualizar tarefa")`)

## Padrão AAA dentro de cada teste
Estruturar cada teste em três blocos separados por linha em branco e comentário:

```java
@Test
void deveCriarTarefaComSucesso() {
    // Arrange
    Tarefa entrada = tarefaValida();
    Tarefa salva = tarefaPersistida(1L);
    when(tarefaRepository.save(any(Tarefa.class))).thenReturn(salva);

    // Act
    Tarefa resultado = tarefaService.criar(entrada);

    // Assert
    assertThat(resultado.getId()).isEqualTo(1L);
    assertThat(resultado.getTitulo()).isEqualTo(entrada.getTitulo());
    verify(tarefaRepository).save(entrada);
}
```

## Detalhes específicos por teste

### Criar
- Mockar `repository.save(...)` para retornar uma tarefa com ID populado
- Verificar que o retorno tem ID e os campos corretos
- Usar `verify(tarefaRepository).save(entrada)` para garantir que foi chamado com o argumento certo
- Opcionalmente usar `ArgumentCaptor<Tarefa>` para inspeção mais detalhada

### Listar
- Caso vazio: `when(repository.findAll()).thenReturn(emptyList())` → assertar lista vazia
- Caso com dados: retornar `List.of(tarefa1, tarefa2)` → assertar tamanho e conteúdo

### Buscar por ID
- Sucesso: `when(repository.findById(1L)).thenReturn(Optional.of(tarefa))` → retorno correto
- Falha: `when(repository.findById(99L)).thenReturn(Optional.empty())` → assertar `TarefaNaoEncontradaException` com `assertThatThrownBy(() -> ...).isInstanceOf(...).hasMessageContaining("99")`

### Atualizar — TESTE CRÍTICO
- **Cenário deve validar preservação de `dataCriacao`**:
  - `Arrange`: criar `tarefaExistente` com `dataCriacao = 2024-01-01T10:00`. Mock `findById` retorna ela. Mock `save` retorna ela mesma (Mockito devolve o argumento via `thenAnswer(inv -> inv.getArgument(0))`)
  - `Act`: chamar `atualizar(id, novosDados)` onde `novosDados` tem outro título mas `dataCriacao` nulo
  - `Assert`: o retorno tem o **título novo** mas **`dataCriacao` original preservada** (`assertThat(resultado.getDataCriacao()).isEqualTo(LocalDateTime.of(2024, 1, 1, 10, 0))`)
- **Cenário de ID inexistente**: `findById` retorna `Optional.empty()` → assertar exceção. Verificar que `save` **nunca foi chamado** com `verify(repository, never()).save(any())`

### Apagar
- Sucesso: `when(repository.existsById(1L)).thenReturn(true)` → chama `deleteById`. Verificar `verify(repository).deleteById(1L)`
- Falha: `when(repository.existsById(99L)).thenReturn(false)` → assertar exceção. Verificar que `deleteById` **nunca foi chamado** com `verify(repository, never()).deleteById(any())`

## Estilo de código
- Java 11 idiomático
- Imports estáticos para `assertThat`, `assertThatThrownBy`, `when`, `verify`, `any`, `never` — testes ficam muito mais legíveis
- Sem JavaDoc nos métodos de teste — o nome do método é a documentação
- Imports sem wildcard, exceto os estáticos do Mockito/AssertJ que são **idiomáticos com wildcard** (`import static org.mockito.Mockito.*` é aceito em testes)
